## Story
**As a** desenvolvedor trabalhando no portfólio,
**I want** que os dados dos projetos sejam carregados a partir de um arquivo `assets/data/projects.json` externo,
**so that** novos projetos possam ser adicionados sem modificar a lógica JavaScript de `projects.js`.

## Acceptance Criteria
1. Criar `assets/data/projects.json` com os 6 projetos atualmente hardcoded em `assets/js/projects.js`.
2. Refatorar `assets/js/projects.js` para fazer fetch assíncrono de `assets/data/projects.json` no construtor da classe `ProjectManager`.
3. Exibir um estado de loading (ex: `.projects-grid { "Carregando projetos..." }`) enquanto o fetch está em andamento.
4. Exibir uma mensagem de erro amigável no `.projects-grid` se o fetch falhar (ex: resposta 404 ou erro de rede).
5. A funcionalidade de filtro por categoria (`.filter-btn`) deve continuar funcionando corretamente após a refatoração.
6. O código do método `createProjectCard` não deve ser alterado — apenas o `loadProjects` e o construtor.

## Dev Notes

### Contexto do Sistema
[Source: docs/brownfield-architecture.md]

- **Arquivo de entrada HTML**: `index.html` — Os projetos são renderizados na `<div class="projects-grid" id="projects-grid">`.
- **Módulo de projetos**: `assets/js/projects.js` — Classe `ProjectManager`. Dados atualmente hardcoded em `this.projects = [...]` dentro do constructor. O método `init()` chama `loadProjects()` e `setupFilters()`.
- **Padrão existente de fetch**: `assets/js/main.js` já utiliza `fetch('assets/data/experience.json')` → Este é o padrão EXATO a ser seguido nesta story.
- **Padrão existente de erro**: O `main.js` faz `timelineContainer.innerHTML = '<p class="error-message">Não foi possível carregar...</p>'` em caso de erro — replicar este padrão.

### Gotcha Técnico
- O construtor `new ProjectManager()` chama `this.init()` de forma direta. Ao tornar `init()` assíncrono, o construtor retorna antes da Promise resolver. Isso é o comportamento **esperado** e **aceitável** para este caso de uso.
- **Não adicionar** `await new ProjectManager()` no event listener — o anti-pattern de `async constructor` não é necessário.

### Estrutura do JSON Alvo
```json
[
  {
    "id": 1,
    "title": "...",
    "description": "...",
    "image": "assets/images/projects/...",
    "category": "web-development",
    "tags": ["Tag1", "Tag2"],
    "codeUrl": "https://github.com/...",
    "demoUrl": "https://..."
  }
]
```

### Mapeamento de Categorias (Filtros HTML existentes)
> ⚠️ O HTML em `index.html` define filtros: `data-filter="all"`, `data-filter="data-analysis"`, `data-filter="machine-learning"`, `data-filter="cloud"`. Mas os projetos hardcoded têm categorias como `"web-development"`. O `@dev` deve preservar as categorias EXATAMENTE como estão no array atual de `projects.js` para não quebrar a lógica de filtro existente (`filterProjects` usa `card.classList.contains(filter)`).
