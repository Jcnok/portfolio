## Story
**As a** proprietário do portfólio,
**I want** que a Vercel Serverless Function `/api/chat` seja mais robusta,
**so that** esteja protegida contra abuso e forneça feedback mais claro ao usuário em caso de erros da API Gemini.

## Acceptance Criteria
1. Adicionar validação de `Content-Type: application/json` na requisição recebida.
2. Capturar e retornar erros específicos da API Gemini (ex: `QUOTA_EXCEEDED`, `SAFETY`, `INVALID_ARGUMENT`).
3. Implementar um rate limit básico por IP (máximo de 15 requisições por minuto) usando a memória da função serverless.
4. O frontend (`assets/js/chat.js`) deve exibir mensagens amigáveis para todos os tipos de erro mapeados.
5. Sanitizar a mensagem do usuário (limitar tamanho, remover caracteres potencialmente perigosos).

### Limitações do Rate Limiting In-Memory
> ⚠️ Vercel Serverless Functions são stateless — cada instância pode ser reciclada a qualquer momento. O rate limiting in-memory (via Map global) funciona dentro da mesma instância quente ("warm start") mas NÃO persiste entre cold starts. Isso é uma limitação **aceitável** para este portfólio pois:
> 1. Oferece proteção contra abuso rápido (burst) na mesma instância
> 2. Não requer infraestrutura externa (Redis, KV store)
> 3. Para um portfólio pessoal, é suficiente como primeira camada de defesa

### Erros Conhecidos da API Gemini
```yaml
RESOURCE_EXHAUSTED: Quota exceeded (free tier limit)
INVALID_ARGUMENT: Malformed request or empty content
PERMISSION_DENIED: Invalid API key
SAFETY: Content filtered by safety settings (returned in finishReason)
INTERNAL: Gemini internal error
```
